**主な推奨事項:**
- ✅ **Zustand** クライアント状態用（UI、フィルター）
- ✅ **SWR** サーバー状態用（タスク、プロジェクト、エリア）
- ✅ **useState** ローカルコンポーネント状態用
- ✅ **Context API** 環境設定のみ

**決定の根拠:**
- Zustand: シンプルなAPI、優れたパフォーマンス、最小限のボイラープレート
- Context APIでない理由: 頻繁な更新でのパフォーマンス問題
- Jotaiでない理由: アトミックモデルが不要な複雑さを追加

## クイックリファレンス

### 技術スタック決定マトリックス

| コンポーネント | 推奨 | 検討した代替案 | 選択理由 |
|-----------|----------------|------------------------|------------|
| **ルーティング** | App Router | Pages Router | Server Components、将来性 |
| **状態管理** | Zustand | Jotai、Context API | シンプル、高パフォーマンス、最小限のボイラープレート |
| **データフェッチング** | SWR | React Query、ネイティブfetch | Vercel製、自動更新に最適 |
| **スタイリング** | Tailwind CSS v4 | CSS Modules、Styled Components | 高速イテレーション、Things3風デザイン |
| **テスト** | Jest + RTL + Playwright | Vitest、Cypress | Next.js組み込みサポート、業界標準 |
| **HTTPクライアント** | ネイティブFetch | Axios | 0KBバンドル、Next.jsキャッシング統合 |
| **アイコン** | Heroicons | Font Awesome、Material Icons | ミニマル美学、Reactコンポーネント |

---

### クリティカル成功要因

1. ✅ **シンプルに始める**: 実証済みツールを使用（最初はJest > Vitest）
2. ✅ **高速イテレーション**: Tailwind + モックモードで迅速なプロトタイピング
3. ✅ **継続的テスト**: 機能と並行してテストを記述
4. ✅ **集中を維持**: 本番環境対応までフェーズ1機能のみ
5. ✅ **コミュニケーション**: PM、アーキテクト、デザイナー、コーダー間の定期的な同期

2. 技術スタックと実装フェーズを承認

1. デザインシステムのため**things3_ui_analysis.md**を学習
2. モックアップで提供されたカラーパレットとタイポグラフィを使用

1. 包括的な理解のためすべてのドキュメントを読む
2. プロジェクトスキャフォールディングのため**nextjs_technical_analysis.md**から開始

1. テスト計画のため**testing_strategy.md**に焦点を当てる
2. APIエラーシナリオのため**api_integration_tailwind.md**を参照
